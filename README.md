# Dekatlas

Rocq を使用して巨大数論を機械化するプロジェクトです。

巨大数論を機械化するというのは、曖昧さがない形で記述して充分に信頼できるソフトウェアに検証させるということです。このことは数学において「形式化する」と呼ぶのが普通なのですが、日常の言葉としての「形式」は「体裁だけ取り繕って中身がない」という良くない意味も含んでいるので、一般の方々にも分かりやすいように本文書では「機械化する」と呼ぶことにします。

## Dekatlas という名前の由来

"Dekatlas" という名前は、巨大数論の地図帳としても、巨大数論を下から支えるプロジェクトとしても、発展してほしいと願い、名づけました。 "dekai" は「でかい」のローマ字綴りであり、巨大数論のことを意味します。 "atlas" は、「地図帳」を意味する英単語であり、ギリシャ神話に登場する天空を下から支えているアトラスという神の名前でもあります。 "dekai" と "atlas" を元にした混成語を作ることで、巨大数論の地図帳という意味と、巨大数論を下から支えるプロジェクトという意味を、二重に持たせています。

## 巨大数論とは

巨大数論とは、巨大数について、あるいは巨大数を生成する表記について研究する学問です。いつ頃から「巨大数論」という言葉があるのかは不明ですが、私の予想では 2 ちゃんねるにおいて巨大数についてのスレッドが立てられるようになった 2002 年頃から使われるようになったのではないかと思います。 P.N. AKIRA SANADA さんが作成した "A CHARMED QUARK" というサイトにおいて「巨大数論」が使われていたという記録が、 2003 年 12 月 26 日のアーカイブ（[リンク](https://web.archive.org/web/20031226011628/http://www.bekkoame.ne.jp/ha/hc17910/short8.htm)）に残っていますから、少なくとも 2003 年頃には使われていたようです。

英語において「巨大数論」と同じ意味の単語は "googology" です。 "googology" は "googol" と "-logy" を組み合わせた言葉です。 "[André Joyce](https://googology.fandom.com/ja/wiki/Andr%C3%A9_Joyce)" (巨大数研究 Wiki) と "[André Joyce](https://googology.fandom.com/wiki/Andr%C3%A9_Joyce)" (Googology Wiki) が説明しているところによると "googology" という言葉を考案したのは Michael Joseph Halm ですが、その事情は込み入っています。

巨大数論が扱う対象は、非常に多様多種であり、巨大数に関わりがあること以外の共通点を見出すことはできません。仏教や娯楽小説などに現れる巨大数を扱うこともあれば、群論やラムゼー理論などに由来する巨大数を扱うこともあります。 well-quasi-ordering や基本列付き順序数表記などを元にして巨大数を作ることもあります。計算不能関数や集合論のモデルを参照して巨大数を定義することもあります。

巨大数論と数論を分かつのは、整数の全体を俯瞰する数論に対して個別の整数に着目する巨大数論という視点の違いであり、巨大数論を数論の下位分野として説明することはできません。そのことは、個別の数に命名するか命名しないかという文化の違いにも表れていますし、数学では問題にならないようなゲーデル数におけるシンボルと素数の対応関係というような細かな定義の違いが巨大数論においては問題にされるということにも表れています。

巨大数論は、現実から離れた巨大な数を扱うことによる応用数学としての意味の薄さと、それぞれの巨大な数に注目することによる純粋数学としての意味の薄さという二律背反に苛まれており、可能な限り大きな数を単純に定義するという工学にも近い性質を帯びていながらも工学が持つべき実用性を持たず、時には大きい数を定義することを目指さない美しい定義を独自性でもって評価するという芸術にも近い性質を帯びていながらも芸術が持つべき自由奔放性を持たない、既存の学問から見捨てられた世界であり、それが故の独自の魅力を有しています。

巨大数の大きさを比較により決定するのは巨大数論の重要なトピックであり、大きさを比較しやすい巨大数が研究対象として好まれ、大きさを比較しにくい巨大数は研究対象として敬遠されます。大きさを比較しやすい巨大数としては、冪乗や矢印表記などのような明快な関数により定義された巨大数があります。大きさを比較しにくい巨大数としては、数学者の Harvey Friedman が考案した関数により定義された巨大数や、ビジービーバー関数のような計算不能関数により定義された巨大数などがあります。

研究対象である巨大数の実在性は巨大数論の重要な要素であるため、現在の巨大数論の大部分は計算可能なアルゴリズムにより定義された巨大数が占めていて、計算不能なアルゴリズムにより定義された巨大数はごくわずかしか研究されておらず、前者と後者の間にある違いが「計算可能巨大数」と「計算不能巨大数」というように呼び分ける形で強く意識されています。ビジービーバー関数のような最も基本的な計算不能関数についても「胡散臭い」と感じるような巨大数論者もいて、数学者の木原貴行は "[ビジービーバー Q & A: 計算不可能の鳥瞰図](https://recursion-theory.blogspot.com/2018/11/q.html)" という記事の中で巨大数論者がビジービーバー関数の存在を受け入れる助けとなるようにという解説をしています。

日本における巨大数論は 2002 年頃に生まれた未成熟な学問であり、その黎明期において大量の間違いが入り込んでおり、その影響は今なお残存していますが、議論が積み重ねられ、 P.N. フィッシュさんが 2013 年に『巨大数論』を上梓して議論の土台を作り、 2018 年ごろから数学者の木原貴行や P.N. p 進大好き bot さんなどが巨大数論における多数の間違いを指摘していき、その結果として巨大数論において厳密な議論が重視されるようになり、その水準は次第に向上しています。

巨大数論に関する書籍は、『巨大数論』（フィッシュ, 2013 年）や、『巨大数』（鈴木真治, 2016 年）や、『数学セミナー 2019 年 7 月号　おおきな数』（日本評論社, 2019 年）や、『現代思想 2019 年 12 月号　巨大数の世界』（青土社, 2019 年）などがありますので、ぜひ目を通してみてください。

## 巨大数論と機械化

P.N. p 進大好き bot さんを始めとした方々による啓蒙が繰り返されたことにより、巨大数論における厳密な議論と経験則に基づいた議論が分離されるようになりました。かつての時代においては、定義がない表記を使用したり、不整合な定義しか持たない表記を使用したりすることが普通にあったので、この点について大きく進歩しています。

しかしながら、依然として、厳密な議論を重視しようにも、それが正しいことを保証する仕組みが薄いという重要な課題が残り続けています。既存の学問から隔絶している巨大数論といえども、数学としての正しさを捨ててしまうと、数学の看板を掲げながら誤った言説を広めることに繋がってしまい、不正義です。かつては、誤った他者への信頼に基づいて、実際には定義が整合していないことが見落とされていた表記が広範に使用されていたりしており、このような正確性がおざなりにされていた時代に逆戻りすることは避けなければなりません。

現在の巨大数論における正確性の検証は、個人に対する信頼に大きく依存していて、特定の個人に負担が集中するだけではなく、その裏返しとして信頼も集中することになり、誤った言説を広める個人を信頼してしまうリスクが高い状況にあります。

通常の分野であれば査読システムがありますが、既存の学問から隔絶している趣味としてしか実行され得ない分野であり、流動性が高い匿名コミュニティに依存している巨大数論において、査読システムは馴染みません。特筆すべき稀有な活動として「東方巨大数」がありますが、数学として検証するのではなく表記の大きさを見積もるだけにとどまっています。

流動性が高い匿名コミュニティに依存している巨大数論において、それでもなお正しさを希求するならば、定理証明支援システムを用いて機械化することが最も良い選択肢です。個人に対する信頼に依存する割合が小さいため、実名であるか匿名であるかの違いが問題にならなくなります。人の出入りが激しくて、個人による貢献が散発的であっても、着実に蓄積されていくことによって、巨大数論の発展に繋がることが期待できます。

"[Determination of the fifth Busy Beaver value](https://arxiv.org/abs/2509.12337)" (The bbchallenge Collaboration, 2025) と "[The Equational Theories Project: Advancing Collaborative Mathematical Research at Scale](https://arxiv.org/abs/2512.07087)" (The Equational Theories Project, 2025) は、定理証明支援システムが通常の数学では考えられないような個人に対する信頼に依存しない大規模な協力を可能にすることを示しており、これらのプロジェクトと同様に巨大数論の機械化もそれぞれの巨大数に対する複数の細かい並列なタスクに分割できるため、同じように定理証明支援システムが有効であることを期待できます。

数学者の Terence Tao は、 "[A pilot project in universal algebra to explore new ways to collaborate and use machine assistance?](https://terrytao.wordpress.com/2024/09/25/a-pilot-project-in-universal-algebra-to-explore-new-ways-to-collaborate-and-use-machine-assistance/)" という記事の中で、数学においては一つの間違いが全体を台無しにしうるため不特定多数や生成 AI からの貢献を受け入れることが難しいという問題を定理証明支援システムが解決し、全体を理解していなくても取り組むことができる細かい並列なタスクに分割できるようなプロジェクトであれば定理証明支援システムによる大規模な協力が可能となることを指摘しています。

巨大数論に定理証明支援システムを導入することにより、より解決が容易になるであろう問題もあります。たとえば、 TREE 数列を解析するのは難易度が高いのですが、その理由は無数の列から最も大きい数を生成する列を見つけ出さなければならないからであり、この点について bbchallenge と類似性があるため、このような形式の問題については定理証明支援システムによる細かい並列なタスクへの分割が有効であるということになります。

## Dekatlas と構成主義数学

Dekatlas は、その大部分が構成主義数学であり、排中律のような非構成的な公理を前提としていません。もちろん、ビジービーバー関数のような必然的に非構成的な対象もあるため、構成主義数学だけというわけではありませんが、そのような対象を扱う場合には非構成的な公理を明示的に仮定しています。

Dekatlas で構成主義数学を用いるのは、通常の数学とは違って個別の整数に注目するため、そのような数について値が確定しないという状況を可能な限り避けたいからです。排中律を使うような普通の数学は存在特性を持たないため、ある巨大数の存在を証明できても当該の値について考察できるようになるとは限りません。数学者の David Hilbert は "[Die Grundlagen Der Mathematik](https://archive.org/details/hilbert-1928-die-grundlagen-der-mathematik)" (David Hilbert, 1928) において "Dieses Tertium non datur dem Mathematiker zu nehmen, wäre etwa, wie wenn man dem Astronomen das Fernrohr oder dem Boxer den Gebrauch der Fäuste untersagen wollte." (p. 16) と書いていますが、この言葉に沿えば「排中律という望遠鏡を覗き込んでばかりいると身近な足元が見えず、排中律という握り拳で殴ってばっかりいると繊細な構造まで砕いてしまうのだ」と例えることができるかもしれません。

巨大数論においては重要である計算可能巨大数と計算不能巨大数の区別という観点からも、構成主義数学における証明は計算可能なプログラムと一致しているため、構成主義数学を土台とすることは理にかなっています。もし、排中律を公理として認めていると、計算可能巨大数を定義しているつもりでも計算不能巨大数になってしまうということが起こるかもしれません。チューリングマシンを基本として計算可能性を証明してもなお、巨大数を計算するアルゴリズムを実装したチューリングマシンの存在性そのものが非構成的に証明されているかもしれず、ビジービーバー関数のグーゴル番目の値を出力するチューリングマシンというような抜け道を許してしまいます。このような事態を避けるためには、全体においては構成主義数学を用いて、特定の区域にだけ非構成的な証明を閉じ込める必要があります。

Dekatlas における構成主義数学は、非構成的な公理を認めず、非構成的な公理と矛盾する公理も認めない中立構成主義数学であり、このことは Dekatlas で証明する定理が全て古典数学における定理でもあることを意味します。中立構成主義数学は古典数学を包み込む形での両立を可能としており、いつでも古典数学に移行することができるため、未来に起こるかもしれない方針の変更に対する頑強性を持っています。数学者の Martín Hötzel Escardó や数学者の Andrej Bauer などが詳しく解説しているように、この意味での構成主義数学とは古典数学の一般化に他ならず、古典数学では見分けられない概念をも見分けることができます。

## Dekatlas と Rocq

Dekatlas は Rocq を使用します。 Rocq は 1989 年にリリースされた定理証明支援システムであり、四色定理の証明や Feit–Thompson theorem の証明などを機械化する際に使われるなど、長年の実績を積み重ねています。

2026 年現在、広く使われているような定理証明支援システムの中で Dekatlas に必要な構成主義数学を行うのに向いているのは Rocq と Agda のみです。 Lean 4 は Rocq と Agda よりも人気がある定理証明支援システムであり、良質なエコシステムと大規模なライブラリを有していますが、 proof-irrevalence を基本としていたり grind タクティックが大域選択公理を前提にしていたりするなど通常の数学を機械化するために調整されているため、構成主義数学を機械化するためには向いていません。

巨大数を真正面から計算することは不可能であるため、巨大数論においては様々な手段で計算量を圧縮しますが、それでもなお巨大なスケールの計算が必要になることがあり、そのような計算を取り扱う能力を Rocq は有しています。たとえば、ビジービーバーチャレンジというプロジェクトは、停止するまで 4717 万 6870 ステップかかるチューリングマシンを Rocq の内部で実行して、それを証明の一部としています。これについては "[Determination of the fifth Busy Beaver value](https://arxiv.org/abs/2509.12337)" (The bbchallenge Collaboration, 2025) という論文に詳細があります。

## Dekatlas に先行するライブラリ

[googology-in-coq](https://github.com/Hexirp/googology-in-coq) は、かつて P.N. Hexirp さんが開発していた構成主義数学において巨大数論を機械化することを目標とするライブラリですが、頻繁な設計の変更が原因で巨大数論を形式化する段階に着手することができていません。

[Googology](https://github.com/choukh/Googology) (choukh) は、かつて P.N. choukh さんが開発していた Rocq を使用して構成主義数学において巨大数論を形式化することを目標とするライブラリであり、ブラウワー順序数を使用して順序数の算術と超限変数ヴェブレン関数を定義しています。

[googology](https://github.com/ccz181078/googology) (ccz181078) は、かつて P.N. ccz181078 さん（またの名を P.N. mxdys さん）が開発していた Rocq を使用して巨大数論を機械化することを目標とするライブラリであり、ブラウワー順序数を使用して順序数の算術と超限変数ヴェブレン関数とブーフホルツ関数を定義しています。構成主義数学を使用しています。

[agda-googology](https://github.com/choukh/agda-googology) (choukh) は、 P.N. choukh さんが開発している Agda を使用して巨大数論を形式化することを目標とするライブラリであり、ブラウワー順序数を使用して順序数の算術と超限変数ヴェブレン関数とブーフホルツ関数を定義しています。構成主義数学を使用しています。

[Coq-BB5](https://github.com/ccz181078/Coq-BB5) は、 P.N. ccz181078 さん（またの名を P.N. mxdys さん）と Tristan Stérin さんが開発した Rocq を使用して `BB(5) = 47,176,870` であることを証明したライブラリです。構成主義数学を使用しています。

Windows の上で Dekatlas を開発するためにインストールする必要があるソフトウェアは次の通りです。

1. Rocq Platform 2025.01.0
2. GNU make
3. Git for Windows

1. Visual Studio Code
2. VsRocq

1. `build_makefile.sh` を実行する。
2. `build.sh` を実行する。
